## 1. Введение

Данный документ описывает план тестирования веб-приложения ["Fly me to the moon"](https://github.com/DanutaGagua/Personal-film-collection-manager). Документ предназначен для команды тестировщиков, участвующих в проекте. Цель тестирования - проверка соответствия реального поведения приложения ожидаемому поведению, описанному в [требованиях](https://github.com/DanutaGagua/Personal-film-collection-manager/blob/master/Documents/Requirements/RequirementsDocument.md).

<a name="items"/>

## 2. Объект тестирования

В качестве объектов тестирования можно выделить следующие функциональные требования:

*   Аутентификация и авторизация пользователей (вход, регистрация, выход).
*   Просмотр доступных авиарейсов.
*   Поиск рейсов (по городу вылета, по городу вылета и прилёта, популярные направления).
*   Бронирование авиабилетов.
*   Просмотр и управление своими бронированиями.
*   CRUD-операции с маршрутами (для владельцев авиакомпаний).
*   Прокладывание пользовательского маршрута.

<a name="quality"/>

## 3. Атрибуты качества

1.  **Функциональность:**
    *   **Функциональная полнота:** Приложение должно выполнять все заявленные функции для всех ролей пользователей (аноним, зарегистрированный пользователь, владелец компании).
    *   **Функциональная корректность:** Приложение должно выполнять все заявленные функции корректно (например, поиск возвращает релевантные результаты, бронирование резервирует место).
    *   **Безопасность:** Доступ к функционалу должен быть разграничен в соответствии с ролями.

2.  **Удобство использования:**
    *   **Эргономичность:** Интерфейс должен быть интуитивно понятным и не перегруженным лишними элементами, соответствовать предоставленным мокапам.
    *   **Доступность:** Время реакции на действия пользователя должно быть минимальным.

3.  **Надёжность:**
    *   **Отказоустойчивость:** Приложение должно корректно обрабатывать некорректные действия пользователя и ошибки ввода данных.

<a name="risk"/>

## 4. Риски

К основным рискам, способным повлиять на процесс тестирования, можно отнести:

*   Нестабильное соединение с базой данных, ведущее к ошибкам при выполнении CRUD-операций.
*   Несовместимость работы приложения с различными версиями браузеров и их настройками.
*   Одновременное бронирование одного и того же рейса разными пользователями (риск "гонки").
*   Некорректная обработка специальных символов и данных на разных языках в полях поиска и при создании маршрутов.

<a name="features"/>

## 5. Аспекты тестирования

В ходе тестирования планируется проверить реализацию основных функций приложения. Аспекты, подвергаемые тестированию:

### 5.1. Работа с учетными записями пользователей
*   **Регистрация нового пользователя.**
    *   Успешная регистрация с корректными данными.
    *   Регистрация с уже существующим логином/email.
    *   Регистрация с некорректными данными (пустые поля, неверный формат email, короткий пароль).
*   **Вход зарегистрированного пользователя в систему.**
    *   Успешный вход с корректными логином и паролем.
    *   Вход с неверным паролем.
    *   Вход с несуществующим логином.
*   **Выход пользователя из системы.**
    *   Корректное завершение сеанса и перенаправление на стартовую страницу.

### 5.2. Функционал для анонимного пользователя
*   **Просмотр доступных рейсов.**
    *   Отображение списка рейсов на главной странице.
*   **Поиск рейсов.**
    *   Поиск по городу вылета.
    *   Поиск по городу вылета и прилёта.
    *   Просмотр популярных направлений.
    *   Поиск с некорректными или несуществующими данными (должен обрабатываться адекватно).
*   **Попытка бронирования без авторизации.**
    *   Система должна перенаправлять на страницу входа.

### 5.3. Функционал для зарегистрированного пользователя
*   **Бронирование авиабилета.**
    *   Успешное бронирование выбранного рейса.
    *   Попытка забронировать недоступный (заполненный) рейс.
*   **Просмотр списка своих бронирований.**
    *   Отображение только бронирований текущего пользователя.
    *   Корректность отображаемой информации о рейсе в бронировании.
*   **Редактирование бронирования.**
    *   Возможность отмены (удаления) бронирования.
*   **Прокладывание собственного маршрута.**
    *   Корректное сохранение пользовательского маршрута на основе выбранных точек.

### 5.4. Функционал для владельца авиакомпании
*   **CRUD-операции с маршрутами.**
    *   **Создание (Create):** Успешное добавление нового маршрута с валидными данными. Ошибка при попытке создать маршрут с некорректными данными.
    *   **Чтение (Read):** Отображение созданных маршрутов в общем списке и в личном кабинете.
    *   **Обновление (Update):** Успешное редактирование информации о существующем маршруте.
    *   **Удаление (Delete):** Успешное удаление маршрута. Проверка, что удаленный маршрут пропадает из поиска и списков.

### 5.5. Нефункциональные аспекты
*   **Удобство использования:**
    *   Соответствие интерфейса предоставленным мокапам.
    *   Интуитивная понятность навигации и элементов управления.
*   **Безопасность:**
    *   Невозможность просмотра или редактирования чужих бронирований путем прямого изменения URL (проверка контроля доступа).
    *   Невозможность доступа к функционалу владельца компании под ролью обычного пользователя.
*   **Производительность:**
    *   Время отклика при поиске рейсов, загрузке страниц должно быть приемлемым (менее 3 секунд).

<a name="approach"/>

## 6. Подходы к тестированию

Для тестирования приложения "Fly me to the moon" будет применяться комбинированный подход:

1.  **Ручное функциональное тестирование:** Все основные сценарии использования (Use Cases) будут протестированы вручную для проверки соответствия функциональным требованиям.
2.  **Тестирование пользовательского интерфейса (UI):** Будут проверяться соответствие макетам, корректность отображения элементов в разных браузерах.
3.  "Дымовое" (Smoke) и "Санитарное" (Sanity) тестирование: Перед каждым крупным этапом тестирования будет проводиться быстрая проверка основных функций на работоспособность.
4.  **Регрессионное тестирование:** После исправления обнаруженных ошибок будет проводиться повторная проверка затронутого функционала.

<a name="pass"/>

## 7. Представление результатов

Результаты тестирования (найденные дефекты, их критическиечность и статус) будут фиксироваться в **Баг-трекинговой системе** (например, Jira, Trello, GitHub Issues). Отчет будет включать в себя:

*   ID и краткое описание дефекта.
*   Шаги для воспроизведения.
*   Ожидаемый и фактический результат.
*   Серьезность дефекта (Blocker, Critical, Major, Minor).
*   Среда тестирования (ОС, браузер, версия).
*   Приложенные доказательства (скриншоты, логи).

Итоговый отчет о тестировании будет сформирован в виде отдельного документа ["Результаты тестирования Fly me to the moon"](https://github.com/DanutaGagua/Personal-film-collection-manager/blob/master/Test%20results.md).

## 8. Выводы

Данный тестовый план охватывает основной функционал приложения "Fly me to the moon". Все критические и блокирующие ошибки должны быть исправлены до окончания тестового цикла.
